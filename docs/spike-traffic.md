# 스파이크 트래픽 대응

대형 마라톤 접수 오픈처럼 분 단위로 트래픽이 수십 배 폭증하는 상황에 대비한 가드레일.

## 원칙
1. **정적 셸은 절대 흔들리지 않는다.** 모든 사용자에게 동일한 HTML이 CDN에서 즉시 응답.
2. **상태 변경은 원자적이고 큐잉 가능.** 결제/접수 같은 쓰기는 사전 입장 큐.
3. **읽기는 어디서든 캐시.** 사용자 무관 데이터는 100% CDN/ISR 캐시.
4. **장애는 격리.** 한 섹션이 죽어도 나머지가 살아 있게 `<Suspense>` + `error.tsx`.

## 사전 준비 체크리스트 (이벤트 24h 전)
- [ ] 트래픽 예상치 산정 (RPS, 동시접속).
- [ ] Fargate HPA min capacity 상향 (스케일 인 안되게 cooldown 길게).
- [ ] CloudFront cache hit ratio 검토 (이벤트 페이지가 캐시되는지).
- [ ] 의존 외부 API의 SLA 확인 + 폴백 준비.
- [ ] DB connection pool 한계 측정 후 컨테이너 max-conns 조정.
- [ ] 부하 테스트(k6/Artillery)로 p95 < 1s 확인.
- [ ] 가입/결제 등 임계 플로우만 카나리 배포로 미리.

## 입장 큐 / 대기실 패턴
- 결제 시작 라우트(`/comp/[id]/checkout`)는 동적이지만 **대기실 셸**(static)로 먼저 진입.
- 대기실에서 토큰 발급 후 실제 결제 페이지로 redirect. 토큰 검증은 엣지 함수에서.
- 동접 한계를 초과하면 대기실에 계속 머무르게 → DB/PG에 부하 안 가게.

## 코드 레벨 가드
- 페이지 컴포넌트는 절대 외부 API 동기 호출 직접 X. 항상 `use cache`로 감싼 함수 경유.
- 사용자 입력 의존 데이터(검색)는 디바운스 + React Query (`staleTime` 길게).
- 무거운 클라이언트 컴포넌트는 `next/dynamic`으로 분리해 메인 페이지 LCP 보호.
- 큰 리스트는 `@tanstack/react-virtual`로 가상화 (메모리/CPU 안정).

## 핫경로 격리
- `/race`, `/community`, `/`는 핫경로. layout/loading.tsx를 가볍게 유지.
- 마이페이지/결제 등 사용자 의존 라우트는 별도 layout 그룹에서 동적 처리. 핫경로 캐시에 영향 X.

## 페일오버
- BFF `/api/*`가 502를 받으면 `{ items: [] }`를 200으로 내려 페이지 깨짐 방지 (현재 구현됨).
- 클라이언트 React Query는 `retry: 1`로 무한 재시도 차단.
- CloudFront SWR(stale-while-revalidate)로 오리진 장애 시 마지막 캐시 응답.

## 사후 검증
- 이벤트 직후 Web Vitals, 5xx, p95, 캐시 적중률 비교.
- 가장 느린 라우트/엔드포인트 식별 → 다음 이벤트 전 개선.
